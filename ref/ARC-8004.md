> **Note on the number.** `8004` mirrors **ERC-8004** for cross-ecosystem clarity; it is **not** an
> officially assigned ARC. This document is the Algorand/AVM port of ERC-8004 *Trustless Agents*
> (Draft, 2025-08-13), produced for the Algorand Builders Berlin hack. Where it diverges from ERC-8004
> the divergence is called out explicitly (see [Relationship to ERC-8004](#relationship-to-erc-8004)).

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT",
"RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as
described in RFC 2119 and RFC 8174.

## Abstract

This standard uses the Algorand ledger to **discover, choose, and interact with agents across
organizational boundaries** without pre-existing trust. It is a construct-by-construct port of
ERC-8004 onto the AVM, defining three Algorand **applications**:

- **Identity Registry** â€” an [ARC-72](https://arc.algorand.foundation/ARCs/arc-0072) smart-contract NFT
  whose `agentURI` resolves to an agent registration file, giving every agent a portable, transferable
  on-chain identity.
- **Reputation Registry** â€” a standard interface for posting and reading feedback signals, tailored so
  each feedback is anchored to an x402 payment on the same ledger.
- **Validation Registry** â€” generic hooks for requesting and recording independent validator checks
  (stake-secured re-execution, zkML verifiers, TEE oracles, trusted judges).

Trust models are pluggable and tiered. Payments remain orthogonal to the registries; this standard
specifies an **x402 profile** that anchors feedback to same-chain payments without the registries ever
gating, verifying, or custodying funds.

## Motivation

Agent communication protocols (MCP, A2A) cover capability advertisement and messaging but not
**discovery** or **trust** across organizations. ERC-8004 answers this with three EVM registries. This
ARC brings the same three registries to Algorand, where they gain native advantages:

- **Boxes** are an arbitrary `bytes`-keyed/`bytes`-valued map â€” a direct, cheaper analog of EVM nested
  `mapping`s, with no per-row contract-storage gymnastics.
- **Native `keccak256`** opcode â†’ `feedbackHash` / `responseHash` / `requestHash` stay byte-identical to
  the EVM standard.
- **Instant finality + sub-cent fees** â†’ an agent-feedback transaction is cheap enough to be a routine
  side effect of doing business.
- **Same-ledger x402 settlement** â†’ "proof of payment" stops being a foreign opaque `{chainId, txHash}`
  and becomes a same-chain, natively-verifiable reference. This ARC leans into that: feedback is
  payment-anchored by construction (see the [x402 Profile](#x402-profile)).

The intended deployment is a **trust router over x402**: an operator requests a service, competing
agents return `402` quotes, the router pays the winner over x402, the work is validated, and the
outcome + payment feed the agent's reputation â€” read back at the next routing decision.

## Specification

### Conventions

Every EVM primitive in ERC-8004 maps to exactly one AVM counterpart:

| ERC-8004 / EVM | This ARC (AVM) | Notes |
|---|---|---|
| Solidity contract | Algorand **Application** (one app id) | 3 contracts â†’ 3 apps |
| `ERC-721` + URIStorage | **ARC-72** smart-contract NFT | programmable transfer hooks (ASAs lack them) |
| `tokenId` (uint256) | `uint64` agentId | incremental counter; never exhausted in practice |
| `mapping(...)` | **box** storage | one box per logical row; key = concatenated tuple |
| Solidity `event` | **ARC-28** event log (`log` opcode) | selector = first 4 bytes of `sha512_256("Name(types)")`; args ARC-4 encoded |

| EVM `address` (20 bytes) | Algorand `address` (32-byte ed25519 pubkey) | all client / validator / wallet / owner fields |
| EIP-712 typed-data sig | **ARC-60** `algorand_signData` + `ed25519verify_bare` | structured-data signing |
| ERC-1271 contract-wallet sig | app-account / logicsig auth | no native 1271 (see [Â§ agentWallet](#on-chain-metadata-and-agentwallet)) |
| EIP-155 `chainId` | **CAIP-2** `algorand:<genesisHashPrefix>` | MainNet = `algorand:wGHE2Pwdvd7S12BL5FaOP20EGYesN73k` |
| `view` function | **ARC-22** readonly ABI method (or `simulate`) | off-chain read, no fee |

All ABI signatures below are ARC-4. Methods marked **(readonly)** are ARC-22 read-only.

### Global agent identifier

Each agent is uniquely identified by:

- **agentRegistry**: `{namespace}:{caip2-ref}:{appId}`, e.g.
  `algorand:wGHE2Pwdvd7S12BL5FaOP20EGYesN73k:1234567`, where the namespace is `algorand` (CAIP-2), the
  chain reference is the genesis-hash prefix, and the registry "address" is the **Identity app id**
  (the app *is* the registry).
- **agentId**: the `uint64` token id assigned incrementally by the Identity Registry.

Off-chain payloads (registration files, feedback files, evidence) MUST include both fields so they can
be tied back to the on-chain agent.

### Identity Registry

A single ARC-72 application holding all agents. `tokenId`â†’`agentId` (uint64), `tokenURI`â†’`agentURI`.

#### State

| Datum | Box / global |
|---|---|

| owner of agentId | box `b"own:"+agentId` â†’ address |

#### Agent URI and registration file

The `agentURI` MUST resolve to the agent registration file. It MAY use any URI scheme (`ipfs://`,
`https://`, or a base64 `data:` URI for fully on-chain metadata). The registration file structure is
**unchanged from ERC-8004**, except `registrations[].agentRegistry` uses the CAIP-2 identifier form:

```jsonc
{
  "type": "https://eips.ethereum.org/EIPS/eip-8004#registration-v1",
  "name": "myAgentName",
  "description": "â€¦",
  "image": "https://example.com/agentimage.png",
  "services": [
    { "name": "A2A", "endpoint": "https://agent.example/.well-known/agent-card.json", "version": "0.3.0" },
    { "name": "MCP", "endpoint": "https://mcp.agent.example/", "version": "2025-06-18" },
    { "name": "algorand-wallet", "endpoint": "<58-char Algorand address>" }
  ],
  "x402Support": true,
  "active": true,
  "registrations": [
    { "agentId": 22, "agentRegistry": "algorand:wGHE2Pwdvd7S12BL5FaOP20EGYesN73k:1234567" }
  ],
  "supportedTrust": ["reputation", "crypto-economic", "tee-attestation"]
}
```

`type`, `services[]`, `x402Support`, `active`, `supportedTrust[]` are chain-agnostic. An agent MAY
advertise its on-chain wallet via an `algorand-wallet` service entry. Endpoint domain verification
(`/.well-known/agent-registration.json`) is transport-layer and unchanged.

#### On-chain metadata and agentWallet

```
getMetadata(uint64 agentId, string metadataKey) -> byte[]      // (readonly)
setMetadata(uint64 agentId, string metadataKey, byte[] value)  // owner/operator; MUST revert if key=="agentWallet"
```

The key `agentWallet` is **reserved**. It is set to the owner address on `register`, MUST NOT be set via
`setMetadata` or the `register` metadata overload, and is **cleared on transfer**. To change it, the
owner MUST prove control of the new wallet:

```
setAgentWallet(uint64 agentId, address newWallet, uint64 deadline, byte[] signature)  // owner/operator
getAgentWallet(uint64 agentId) -> address    // (readonly)
unsetAgentWallet(uint64 agentId)             // owner/operator
```

- **EOA equivalent (ed25519 keypair).** `newWallet` signs an ARC-60 structured payload
  `{appId, agentId, newWallet, deadline}`, domain-separated with the genesis hash to prevent cross-app /
  cross-network replay. The app verifies with `ed25519verify_bare`. Since an Algorand address *is* the
  ed25519 pubkey, this is a clean 1:1 for the EOA case.
- **Contract-wallet equivalent (ERC-1271 has no native analog).** The new wallet (an app account)
  authorizes by issuing an inner-transaction app-call to `setAgentWallet`; `Txn.Sender` IS the proof.
  (A logicsig variant using an in-group 0-amount self-payment inspected via `gtxn` is also permitted.)

`deadline` MUST satisfy `Global.LatestTimestamp <= deadline <= Global.LatestTimestamp + 5 minutes`.

#### Registration

```
register(string agentURI, (string,byte[])[] metadata) -> uint64   // mints, sets uri, sets agentWallet=caller
register(string agentURI) -> uint64
register() -> uint64                                               // agentURI added later
setAgentURI(uint64 agentId, string newURI)                        // owner/operator
isAuthorizedOrOwner(address spender, uint64 agentId) -> bool      // (readonly) â€” used by other registries
```

#### Events (ARC-28)

```

```

> **Indexed-topic note.** EVM `indexed` params become Ethereum log topics. AVM `log` has no topics; an

| Datum | Box |
|---|---|
| per-(agent,client) last index | `b"li:"+agentId+client` â†’ uint64 |
| feedback row | `b"fb:"+agentId+client+index` â†’ ARC-4 `(byte[16] value, uint8 dec, string tag1, string tag2, bool isRevoked)` |
| client set per agent | `b"cl:"+agentId` â†’ address[] (append on first feedback) |
| response count | `b"rc:"+agentId+client+index+responder` â†’ uint64 |

`endpoint`, `feedbackURI`, `feedbackHash` are **emitted, not stored** (1:1 with ERC-8004). `keccak256`

#### Giving feedback (x402-anchored â€” see [x402 Profile](#x402-profile))

```
giveFeedback(
  uint64  agentId,
  byte[16] value,            // int128 two's-complement big-endian
  uint8   valueDecimals,     // 0..18
  string  tag1,
  string  tag2,
  string  endpoint,
  string  feedbackURI,
  byte[32] feedbackHash,
  byte[32] paymentTxid,      // MANDATORY â€” x402 settlement txid on this ledger
  uint64  nonce              // MANDATORY â€” x402 challenge nonce
)
```

Checks:

- `agentId` MUST be a validly registered agent. The Reputation app reads the Identity app's
  `b"own:"+agentId` box (foreign app + box reference in the txn) to confirm existence.
- `valueDecimals` MUST be in `[0,18]`; `value` MUST satisfy `|value| <= 1e38`.
- The caller MUST NOT be the owner or an approved operator of `agentId` (self-feedback prohibition),
  checked via `isAuthorizedOrOwner` on the Identity app.
- `paymentTxid` and `nonce` **MUST be present** (`paymentTxid != bytes32(0)`). The contract records and
  emits them. **The contract MUST NOT verify the settlement on-chain** â€” see the x402 Profile for the
  rationale (mandatory *reference*, not on-chain *enforcement*).

The stored row is `{value, valueDecimals, tag1, tag2, isRevoked}`; `feedbackIndex` is a 1-indexed
counter per `(agentId, clientAddress)`.

#### Interpreting value + valueDecimals

Treat the pair as a signed decimal: `value=9977, valueDecimals=2 â†’ 99.77`; `value=560, valueDecimals=0
â†’ 560`. The meaning is conveyed by `tag1`/`tag2` and/or the off-chain file (percentages, scores,
timings, amounts). x402 money signals (ÂµALGO / USDC base units) fit `uint64`; implementations SHOULD use
the unsigned fast-path for these and defer signed aggregation off-chain.

### x402 Profile

This ARC is tailored for x402 agentic commerce. The profile is **normative for the Reputation Registry**
(it makes the `giveFeedback` payment fields mandatory) and **conventional elsewhere**. Its guiding line:
**require the payment reference; do not verify, gate, or move the funds.** Payments stay orthogonal to
the registries, exactly as ERC-8004 intends ("payments are out of scope; x402 only enriches feedback").

The one AVM-native lever: x402 settles on the **same Algorand ledger** as the registries, so a payment
reference is a same-chain, natively-verifiable pointer rather than a foreign opaque hash.

#### 1. Mandatory payment anchor on `giveFeedback`

`giveFeedback` REQUIRES `paymentTxid` (`byte[32]`, the x402 settlement txid on this ledger) and `nonce`
(`uint64`, the x402 challenge nonce). The contract MUST `require` them present and MUST record + emit
them in `NewFeedback`. The contract MUST NOT verify the settlement on-chain.

This is the profile's single **deliberate divergence** from ERC-8004's "all fields except value /
valueDecimals are OPTIONAL." Rationale: for the trust router an unpaid review is meaningless, so a
payment anchor is a precondition of feedback â€” but on-chain payment *enforcement* (gating, escrow,
`gtxn` settlement checks) would break ERC-8004's orthogonality. Mandatory **reference**, not on-chain
**enforcement**.

#### 2. Same-chain proof-of-payment (off-chain file)

The off-chain feedback file's `proofOfPayment` is specialized from ERC-8004's
`{fromAddress, toAddress, chainId, txHash}` to same-ledger form:

```jsonc
"proofOfPayment": {
  "from": "<payer address>",
  "to": "<agent wallet address>",
  "asset": 0,            // 0 = ALGO, else ASA id (e.g. USDC)
  "amount": 1000000,     // base units
  "txid": "<same-chain settlement txid>",
  "round": 12345678,
  "nonce": 42            // x402 challenge nonce
}
```

`{from, to, asset, amount, round}` are resolvable from `txid` on the same chain, so they are
informational; the on-chain mandatory anchor is `paymentTxid` + `nonce`.

#### 3. Tag and client conventions

- `tag1 = "x402"` marks payment-backed feedback; `tag = "x402:settled"` (and finality stages) marks
  validation context. `tag1`/`tag2`/`tag` are developer-discretion per the base spec.
- `value`/`valueDecimals` MAY carry the paid amount as a money signal.
- `clientAddress` SHOULD be the x402 payer address. **Verified-purchaser reputation** then emerges from
  the stock read `getSummary(agentId, payers[], "x402", â€¦)` â€” payment filters reputation at *read* time;
  the contract never requires payment validity at *write* time.

#### 4. Router correlation

The off-chain x402 `nonce`/`txid` is the **correlation key** an indexer uses to stitch
payment â†” validation â†” feedback. This is read-side only; nothing on-chain requires it. `nonce` is
recorded for the indexer's correlation/dedup, **not** enforced by the contract.

#### 5. Non-goals (these would break ERC spirit)

- The registries MUST NOT perform on-chain settlement **verification** (`gtxn` inspection of the
  payment). The x402 facilitator verifies settlement off-chain.
- The registries MUST NOT **escrow**, custody, or release funds via inner transactions.
- The registries MUST NOT enforce payment **anti-replay**; the facilitator owns replay protection.
- Apart from the mandatory `giveFeedback` anchor, x402 fields are OPTIONAL.

### Gas sponsorship (fee pooling)

A relayer MAY add itself to the atomic group and pay the client's `giveFeedback` fee via fee pooling,
so the client needs 0 ALGO. This realizes ERC-8004's EIP-7702 gas-sponsorship rationale and works today
without account abstraction.

## Rationale

- **Three apps, not a monolith.** Mirroring ERC-8004's three registries keeps Identity reusable across
  trust models and lets Reputation/Validation read Identity via foreign-app box references.
- **Boxes over global state.** Per-row boxes give unbounded, cheap, arbitrary-width storage matching the
  EVM nested-mapping model; global state holds only counters and the cross-registry app id.
- **Aggregation off-chain.** AVM has a bounded opcode budget; unbounded read loops do not port. Emitting
  ARC-28 events and aggregating off-chain is both AVM-idiomatic and aligned with ERC-8004's "complex
  aggregation happens off-chain."
- **Same-chain x402 anchor.** Because settlement and reputation share a ledger, a mandatory `paymentTxid`
  is verifiable by any indexer without cross-chain trust â€” enabling verified-purchaser reputation while
  keeping the contract payment-agnostic.

## Relationship to ERC-8004

This ARC is a faithful port. The genuine divergences are called out honestly:

1. **No indexed event topics.** AVM `log` has no topics; ARC-28 emits whole events and indexers filter
   decoded fields. Functionally equivalent, structurally different.
2. **ERC-1271 has no native analog.** Contract-wallet signature validation in `setAgentWallet` is
   re-expressed as method-call (inner-txn) or logicsig auth. The EOA/EIP-712 case ports cleanly via
   ARC-60 + `ed25519verify_bare`.
3. **uint256 â†’ uint64 agentId.** ARC-72 nominally uses uint256 token ids; this ARC constrains the
   *agentId counter* to uint64. `feedbackHash`/`responseHash`/`requestHash` keep the full 32 bytes.
4. **int128 signed math.** `value` ports as `byte[16]` two's-complement; on-chain aggregation needs
   explicit sign handling, or (recommended) defers off-chain. Storage and emission are 1:1.
5. **Mandatory x402 anchor on `giveFeedback`** (this ARC only). ERC-8004 makes all fields except
   `value`/`valueDecimals` optional; this ARC's x402 profile makes `paymentTxid` + `nonce` mandatory on
   `giveFeedback` (recorded + emitted, **not** on-chain-verified). Strip the profile to obtain a vanilla
   ERC-8004-equivalent registry.

Everything else â€” three registries, all functions, all events, all hashes (keccak-256 native),
identity-via-NFT, URIStorage, metadata kv, reserved `agentWallet` with transfer-clear semantics,
self-feedback prohibition, Sybil guard on `getSummary`, progressive validation â€” ports 1:1.

## Security Considerations

- **Sybil attacks** can inflate fake-agent reputation. This standard makes signals public and uniform;
  the mandatory x402 anchor plus reviewer filtering (`getSummary(clients[])`) raises the cost of fake
  positive reputation, but does not eliminate Sybil risk. Reputation systems over reviewers are expected
  to emerge.
- **Payment references are unverified on-chain.** `paymentTxid` is mandatory but not checked by the
  contract; an indexer/facilitator MUST confirm the txid actually settled to the agent's wallet before
  trusting "verified-purchaser" status. This is a deliberate trade to keep payments orthogonal.
- **Audit-trail integrity.** Boxes are mutable by app logic, but ARC-28 logs live in the immutable
  ledger, preserving an audit trail exactly as EVM event logs do.
- **Signature replay.** `setAgentWallet` payloads MUST be domain-separated with `appId` and genesis hash
  and bounded by `deadline` to prevent cross-app/cross-network replay.
- **Validator incentives and slashing** are managed by specific validation protocols, outside this ARC.
- This standard cryptographically ties the registration file to the on-chain agent but cannot guarantee
  advertised capabilities are functional or non-malicious; the trust models address that need.
